## Project Context

### Steering vs Specification

- Steering(@./.kiro/steering/): Guide AI with project-wide rules and context
- Specs(@./.kiro/specs/): Formalize development process for individual features

### Active Specifications

- Check @./.kiro/specs/ for active specifications
- Use `/asdd:status {feature-name}` to check progress

## Workflow

### Steering (Optional)

- `/asdd:steering` - Create or update steering documents

Note: Optional for new features or small additions. Can proceed directly to init.

### Specification

1. `/asdd:init {detailed description}` - Generate spec.md with intent, requirements, approach, and rules

### Implementation

2. `/asdd:impl {feature} [req-ids] [-y]` - Execute TDD implementation (approval gate on first run; `-y` to skip)

### Evolution

3. `/asdd:evolve {feature}` - Update spec when requirements change during implementation

### Progress

- `/asdd:status {feature}` - Check current progress

## Development Rules

1. **Single spec file**: Each feature has one spec.md (not 3 separate files)
2. **Approval gate**: First `/asdd:impl` run asks for spec review confirmation; use `-y` only for intentional fast-track
3. **TDD mandatory**: Implementation follows strict Red-Green-Refactor cycle
4. **Blocker detection**: AI reports evidence-based blockers during implementation
5. **Human approval**: spec.md changes require explicit human approval
6. **Prefix matching**: Feature names support prefix matching in all commands
7. **Keep steering current**: Run `/asdd:steering` after significant changes

## Steering Configuration

### Core Steering Files (generated by `/asdd:steering`)

- `product.md`: Product context and business objectives
- `tech.md`: Technology stack and architectural decisions
- `structure.md`: File organization and code patterns
